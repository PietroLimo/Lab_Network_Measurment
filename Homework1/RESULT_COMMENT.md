# Homework 1 - Active Measurements

## Goal

The goal of this assignment is to evaluate the relationship between the average RTT and the physical distance between two endpoints.

The first endpoint is my machine, where the Python script is executed.
The second endpoint is a set of remote servers distributed across different geographic locations.

The assignment requires:

- estimating the average RTT to multiple remote hosts,
- computing the physical distance in km between the source and each destination,
- collecting both short-distance and long-distance measurements,
- plotting RTT versus distance,
- fitting a linear model to estimate the average RTT per km,
- commenting on the obtained result.

## Files

- `rtt_distance.py`: main script used to perform the measurements and generate the plot.
- `server.csv`: list of destination servers with hostname and geographic coordinates.
- `results.csv`: output file containing the measured RTT values and computed distances.
- `rtt_vs_distance.png`: plot generated by the script.
- `1 - active_measurements(1).pptx`: lecture slides containing the assignment.

## Approach

The script performs the following steps:

1. It loads the list of servers from `server.csv`.
2. It determines the source coordinates, either from command-line arguments or through IP-based geolocation.
3. For each server, it sends ICMP echo requests using the system `ping` command.
4. It extracts the average RTT from the `ping` output.
5. It computes the geographical distance using `geopy.distance.distance()`.
6. It stores all results in `results.csv`.
7. It fits a linear model of the form:

   `RTT = m * distance + q`

8. It plots the measured points together with the fitted line.
9. It saves the plot as `rtt_vs_distance.png`.

## Libraries Used

The script uses the following Python libraries:

- `numpy`
- `pandas`
- `matplotlib`
- `geopy`
- `requests`

These libraries are allowed for the homework and are used only to automate the measurements, data handling, fitting, and plotting.

## How To Run

Example:

```bash
python3 rtt_distance.py --servers server.csv
```

Optional parameters:

```bash
python3 rtt_distance.py \
  --servers server.csv \
  --src-lat <latitude> \
  --src-lon <longitude> \
  --count 10 \
  --timeout 2 \
  --sleep 0.2 \
  --out results.csv \
  --plot-out rtt_vs_distance.png
```

## Comment On The Result

Using the current `results.csv`, the fitted linear model is approximately:

`RTT = 0.0203 * distance_km + 27.15`

with:

- slope `m = 0.0203 ms/km`
- intercept `q = 27.15 ms`
- coefficient of determination `R^2 = 0.8907`

This result shows a clear positive correlation between RTT and physical distance: in general, RTT increases as the destination becomes farther away.

The slope means that, on average, the RTT increases by about `0.0203 ms` for every additional kilometer of distance. Equivalently, this is about `20.3 ms` every `1000 km`.

The intercept is not zero because RTT is not determined only by propagation delay. It also includes:

- processing delay in end hosts and routers,
- queueing delay,
- protocol and operating system overhead,
- path inefficiencies and routing policies.

The `R^2` value is relatively high, so the linear model explains most of the observed trend. However, the points are not perfectly aligned, which is expected because RTT depends not only on geographic distance but also on congestion, routing asymmetry, intermediate hops, and server/network conditions.

For example, some European servers show RTT values that are lower or higher than other servers at similar distances. This confirms that physical distance is an important factor, but it is not the only one.

Overall, the measurements are consistent with the expected theory: longer distances tend to produce larger RTT values, and a linear approximation provides a reasonable first-order model.

## Technical Notes

- The script parses the output of the local `ping` command, so it works on Unix-like systems such as macOS and Linux.
- If source coordinates are not manually specified, the script uses an IP geolocation service to estimate the source position.
- Failed pings are still recorded in the CSV, but only valid RTT values are used for the fitting step.
